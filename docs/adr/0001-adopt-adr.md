# ADR-0001: Architecture Decision Recordsの導入

## コンテキスト

GitHub Actionsで使用するReusable Workflowを定義する上でyamlファイル内でのインラインのShellScriptでは限界がある。
そのため、リポジトリ内にWorkflowで使用するスクリプトを配置して呼び出すことにした。
スクリプトに使用するプログラミング言語やライブラリ等の選定の基準をドキュメントとして残すことで未来の私やCoding Agentに対して、
何をどういう経緯で採用したかを伝える必要がある。

## 検討した選択肢

### 選択肢1: MADR（Markdown Any Decision Records）

軽量なMarkdownベースのフォーマット。
GitHubでの閲覧に適しており、PRレビューのワークフローに馴染む。

#### 良い点

- シンプルで導入コストが低い
- Markdownなのでツール不要

#### 悪い点

- 厳密なスキーマがないため記述にばらつきが出る可能性がある

### 選択肢2: RFCプロセス

設計提案を正式なRFCとして管理する。

#### 良い点

- 大規模な変更に対して十分な議論の場を提供できる

#### 悪い点

- 個人開発の小規模プロジェクトにはオーバーヘッドが大きい

### 選択肢3: Wikiやissueでの記録

GitHubのWikiやissueに設計判断を記録する。

#### 良い点

- 追加のファイル管理が不要

#### 悪い点

- コードと一緒にバージョン管理されない
- 検索性が低い

## 決定

MADR形式のADRを`docs/adr/`に配置する。

本プロジェクトは個人開発の小規模CLIツールであるため、軽量なフォーマットが適している。
Markdownファイルとしてリポジトリに含めることでコードと一緒にバージョン管理でき、PRレビューの対象にもなる。

## 結果

### 良い影響

- 設計判断の「なぜ」が明示的に記録され、将来の判断の参考になる

### 悪い影響

- ADRの作成・維持にわずかなコストが発生する
